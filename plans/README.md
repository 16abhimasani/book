## Findings considered and rejected (do not re-audit)

- **"CRITICAL: stop ≥ entry reports $0 risk" (as a bug):** REJECTED as framed.
  `risk.ts:105` `Math.max(0, (entry-stop)*qty)` is correct — a position whose
  stop has ratcheted above entry (profit-locking trailing stop, e.g. INTC
  entry 114.15 / stop 116.34) genuinely has zero downside-to-entry. Clamping to
  $0 is by-design. The *real* kernel (a typo and a profit-lock look identical)
  survives as **plan 004** (an informational flag, not a failed check).
- **Per-lane §6a stats not wired into `bun run book`:** REJECTED — already done.
  `book.ts:89` calls `computeLaneStats` inside `assemblePanel`. Subagent had
  stale info.
- **CSV formula injection (prepend `'` to `=+@-`):** Not worth doing now. The
  CSVs are machine-read/written only; no spreadsheet workflow exists. Revisit
  only if a "export to analyst CSV" feature is added.
- **Shadow N+1 / gate MA caching (performance):** Not worth doing. Data is ~45
  marks rows and a handful of trades; these are O(n) on tiny n. Micro-opt.
- **Single `status` entry point:** Not worth a new file — `bun run book` already
  is the consolidated panel (positions, stops, §2 limits, gate, per-lane §6a,
  staleness/missing-stop flags).
- **Heartbeat mutex / POLICY checksum / push-failure handling / dep audit:**
  Real but out of scope for these code plans — they are operational/owner-domain
  (OPERATIONS.md), and the existing git-ordering mitigation plus the owner's
  one-heartbeat decision cover the live risk. Track in OPERATIONS, not here.
